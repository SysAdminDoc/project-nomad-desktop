# `get_db()` leak audit — factory-loop iteration 2

Total suspect call sites: **34** across 14 files.

Generated by `tools/audit_db_sessions.py`. A finding here means a `get_db()` call isn't paired with either a `db_session()` context manager (preferred) or a `try/finally: db.close()` pair. Each is a candidate pooled-connection leak.

## Severity legend

- **assigned-but-not-closed** — `db = get_db()` without a `try/finally` that closes `db`. Highest priority; the worker holds a live connection for the caller's lifetime.
- **bare-call** — `get_db().execute(...)` as an expression. The connection is abandoned after the chained call; the pool will eventually reclaim it but transactions may linger.
